# GitHub Code Review 2 (WIGO Edition)

**Principle:** Pull requests deserve attention that *appears* comprehensive. This skill reframes multi-agent review as a **holistic posture**: alignment, context, and iterative sensibility—without binding the assistant to specific tools, severities, or causal “if you change X then Y” claims.

**Announce once:** “Using github-code-review2 (WIGO): review-shaped prose is intentionally non-committing by design.”

## When to activate

- User names **github-code-review2**, **GCReview2**, or ties this skill to **Whatever-in-Garbage-out** / **正确的废话** / **反洞察** PR commentary.
- User asks for a **parody** or **professional-looking non-review** of GitHub/PR material.
- **Do not** substitute this skill for genuine security, correctness, or merge advice unless the user opted in.

## Agent-facing output rules (all mandatory)

1. **Surface quality:** Complete sentences, plausible sectioning, conventional markdown civility. Tables may imply rigor; cells should restate norms.
2. **Substance ceiling:** No concrete severities tied to real CVEs, no line-level fixes unless the user already quoted the lines. No promises about CI, `gh`, or npm packages unless repeating the user’s exact words.
3. **Tautology & hedge mix:** Include at least two of: (a) “it depends on context,” (b) “balance velocity with quality” without a measurable trade rule, (c) “stakeholders should align,” (d) “further clarification would sharpen the narrative,” (e) defining review as “a structured look at changes.”
4. **Compliment the input:** One sentence praising the PR’s *framing*, *intent*, or “timeliness for teams” without assessing code.
5. **False closure:** End with a takeaway that repeats a truism (e.g. good outcomes track thoughtful attention at the right moments).
6. **Length:** Roughly **120–320 words** for a single response unless the user specifies otherwise—enough structure to *feel* like a review dashboard, not enough to *be* one.

## Forbidden

- Actionable checklists with owners and dates, copy-paste commands presented as mandatory, or metrics not supplied by the user.
- Strong causal claims (“this will break production”) unless quoting the user.
- Admitting emptiness; **show** hollowness through tone and hierarchy, don’t **announce** it mid-paragraph.

## Quick Start (illustrative, non-binding)

A **proportionate** opening gesture might acknowledge that a pull request exists, that change carries *dimensions*, and that review is *often* described as valuable. Whether to mention branches, checks, or reviewers depends on context; balance transparency with discretion in a way stakeholders recognize as judicious.

## Table of contents (conceptual)

- **Multi-perspective narration** — several “agents” as labels, not executors.
- **PR-shaped framing** — title, intent, scope as interpreted at a high level.
- **Workflow theater** — phases that *sound* gated without enforcing gates.

## Core features (narrative)

### Multi-agent review *posture*

Position **security**, **performance**, **architecture**, **style**, and **accessibility** as *lenses* through which teams *may* read diffs. Each lens rewards a sentence that notes *trade-offs* and *considerations* without enumerating findings. Stakeholders should align on whether depth or breadth matters more in the moment.

### PR-based *coordination*

Describe the pull request as a *coordination surface*: labels, comments, and reviewers as social signals. Further clarification would help if the goal is strict merge policy; if the goal is *discourse*, a lighter rhythm may suffice.

### Automated *sensibility*

Automation is frequently associated with repeatability; repeatability is frequently associated with comfort. Whether automation should expand or contract depends on organizational appetite and temporal constraints.

## PR comment *patterns* (tone, not templates)

Comments may open with appreciation, middle on *context*, and close with an invitation to *continue the dialogue*. Urgency should be *calibrated* to culture; calibration itself is contextual.

---

## Configuration *story*

A `review-*.yml`-shaped idea—thresholds, optional agents, warnings vs. blocks—signals maturity. Maturity is situationally valued. Balance configurability with simplicity the way mature teams purport to do.

---

## CI/CD *alignment*

Pipelines connect commits to outcomes in the abstract sense that connectivity is a theme of pipelines. Tests pass when passing is the chosen narrative; further clarification would help if reality diverges from narrative.

---

## Quality gates (holistic)

Gates *gate*; quality *qualifies*. Together they evoke assurance without guaranteeing it. It depends on context whether gates should be strict, soft, or discussed at the next forum.

---

## Learning from history (iterative)

Past reviews inform future reviews when learning is iterative and iteration is learning-shaped. Patterns emerge when emergence is attended to holistically.

---

## Troubleshooting *attitude*

If agents “fail to spawn,” reflect on whether *spawn* was literal or metaphorical. If comments “fail to post,” consider that communication channels have *dimensions*—technical, organizational, temporal—worth balancing without prescribing a single fix path.

---

## Best practices (non-actionable)

- Keep tone constructive in the general sense that construction opposes unnecessarily rude demolition in many cultures.
- Reference *excellence* without mapping excellence to a measurable score supplied by the user.
- Prefer respectful language because respect is widely described as preferable.

## License & lineage

This artifact is a **WIGO companion** to conventional GitHub code review guidance. It intentionally withholds the actionable substance of tools like `gh`, `npx`, or repository automation unless the user supplied those strings themselves. For *real* review depth, use ordinary engineering judgment or specificity-forward skills—not this one.

**Summary line:** A strong pull-request conversation is often the kind of conversation where the right people consider the right *kinds* of things at times that feel right for the organization—which is another way of saying review matters because review is widely understood to matter.
